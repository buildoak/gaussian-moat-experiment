## Task: Produce an alignment plan for tile-validator Python code against authoritative specs

### Context
You are in the `tiles-maxxing/` directory of the gaussian-moat-cuda project. There is a Python tile validator at `tile-validator/` that was written as a naive first implementation. Since then, three authoritative specs have been written and refined:

- `docs/tile_spec.md` — tile geometry, coordinate system, collar, sieve domain
- `docs/tile_operations.md` — 5-phase pipeline, TileOp encoding, port structure, group semantics
- `docs/compositor_spec.md` — tile composition, seam merging (read if relevant to validator)

The specs are the single source of truth (see AGENTS.md). The Python code predates them and may diverge.

The goal: Python is fast to iterate on. We want tile-validator to be a spec-faithful reference implementation so we can use it to validate C++ and CUDA outputs with confidence. If the Python code disagrees with the specs, Python must change.

### What to do

**Phase 1 — Read everything.** Read all Python files in `tile-validator/` thoroughly:
- validate.py
- tile.py
- ports.py
- compose.py
- Any other .py files present

Also read the three specs cover to cover:
- docs/tile_spec.md
- docs/tile_operations.md

You need to understand both sides completely before comparing.

**Phase 2 — Diff against specs.** For each phase of the 5-phase pipeline (sieve, compact, union-find, face-extract, encode), compare:

1. Does the Python implement this phase?
2. If yes, does the algorithm match the spec exactly?
3. If no, what is missing?
4. For each divergence: what exactly does Python do vs what the spec says?

Pay special attention to:
- Coordinate conventions (bottom-left corner? center? 0-indexed?)
- Collar size and sieve window dimensions
- How primes are identified (norm primality test method)
- Union-find connectivity: step distance definition (Euclidean? taxicab? norm-based?)
- Face extraction: which primes are boundary primes? How are ports defined?
- Port clustering: distance-based vs shared-prime matching
- TileOp encoding: does Python produce the 128-byte format? Or something else?
- Group numbering: 0-indexed? 1-indexed? Overflow sentinel handling?
- Pruning: dead-end pruning logic, when ports are removed

**Phase 3 — Produce the plan.** Output a structured execution plan:

For each divergence found, create a work item:
```
### [priority: critical/high/medium/low] Short title
- **Spec says:** <exact quote or reference with section number>
- **Python does:** <what the code actually does, with file:line references>
- **Gap:** <precise description of the mismatch>
- **Fix:** <what needs to change in Python, concretely>
- **Risk if unfixed:** <what goes wrong if we leave this>
```

Order work items by priority (critical first).

Also note:
- Things the Python code does CORRECTLY per spec (confidence builders)
- Things the spec is silent on that Python implements (potential spec gaps to flag to user)
- Any constants or magic numbers in Python that should be derived from spec parameters

### Output format
A single structured document suitable for saving to `docs/supportive/2026-04-09-validator-alignment-plan.md`. Include frontmatter:
```yaml
---
title: Tile Validator Alignment Plan
date: 2026-04-09
engine: codex
type: analysis
status: complete
refs: [docs/tile_spec.md, docs/tile_operations.md, tile-validator/]
---
```

Be thorough but not verbose. Every claim must reference a specific line in the code or section in the spec.
