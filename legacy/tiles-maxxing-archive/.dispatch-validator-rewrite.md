### Alignment plan (7 ORDERED work items from the alignment plan at docs/supportive/2026-04-09-validator-alignment-plan.md)

Read the full alignment plan first. Then execute these steps in order:

**Step 1: Fix coordinate conventions**
- Rename "top-left" to bottom-left origin convention
- Use `a_lo/b_lo` or `x_lo/y_lo` consistently
- Fix face naming: I=bottom, O=top, L=left, R=right per spec
- Update all files that reference coordinates

**Step 2: Add compact phase**
- After sieve, build bitmap (one bit per lattice point, 1=prime)
- Build prefix popcount table
- Build dense `prime_pos` array mapping compact index to (a, b) position
- Create a new module or add to existing tile.py
- This is Phase 2 in the spec pipeline

**Step 3: Replace union-find with spec algorithm**
- Spec requires: NO rank array, smaller-index root always wins, full path flattening
- Replace rank-based UF in uf.py with spec-faithful version
- Rebuild Phase 3 on compacted prime indices (not raw coordinate tuples)
- Neighbor lookup uses backward offset table within distance sqrt(k)

**Step 4: Rewrite face extraction and port clustering**
- CRITICAL: face primes are INSIDE the tile proper, within collar distance of boundary
- Halo primes are NOT face primes (spec is explicit about this)
- Port clustering uses `distance^2 <= k` (2D Euclidean, both along-face and depth)
- NOT the old 1D along-face separation <= 6
- Sort face primes by h-coordinate, scan with sliding window checking distance^2 <= k
- Each cluster = one port, labeled with its component's group ID

**Step 5: Add dead-end pruning, 1-based groups, overflow sentinels**
- Group labels start at 1 (0 is reserved for empty slots)
- Dead-end pruning: omit any group that touches exactly one face with exactly one port on that face
- After pruning, renumber groups contiguously starting at 1
- Overflow check: if any face has > 16 ports after pruning, or group_count > 255, the tile overflows
- On overflow: entire 128-byte TileOp becomes 0xFF (all bytes)

**Step 6: Emit real 128-byte TileOps**
- Implement the spec byte layout from tile_spec.md S4.1 and tile_operations.md S8.1
- Bytes 0-63: I/O/L/R face group labels (16 bytes per face, 0x00 = empty slot)
- Bytes 64-95: L/R h1 values (exact layout per spec)
- Bytes 96-127: reserved, zero-filled
- Overflow sentinel: all 128 bytes = 0xFF
- The TileOp bytes must be the canonical output of process_tile()

**Step 7: Fix composition and validation**
- Composition works on 128-byte TileOps, not Python objects with fingerprints
- I/O face matching: by slot index after zero-sentinel alignment
- L/R face matching: by h1 + delta_h per spec
- Fingerprint matching becomes optional diagnostic, not the primary mechanism
- Update validate.py to test against spec outputs (byte-level TileOp correctness)
