## Platforms

### CUDA — RTX 4090 (vast.ai)

- SM architecture: sm_89, 128 SMs, 24 GB VRAM
- Multi-kernel pipeline: K1 sieve → K2 MR → K3 compact → K4 UF → K5 face-encode
- K2 config: maxrregcount=44, 288 threads/block
- MR variant: FJ64_262k, 2-round Miller-Rabin

### C++ Reference — M4 Mac Mini

- 12-core parallel execution via parallel_cpp_dump.py
- 128-bit MR primality testing
- Used as ground-truth validator

## Cross-Validation

**Result: 300,000 / 300,000 EXACT match**

All 300K tiles produce bit-for-bit identical 128-byte TileOps between CUDA and C++. Zero mismatches. Zero permutation-equivalent differences. The CUDA multi-kernel pipeline is fully validated against the C++ reference at R=850M.

**Known legitimate divergence:** `prime_count` differs between platforms. CUDA counts sieve-domain primes including the 7-unit collar (271×271 domain), C++ counts tile-proper primes in the 257×257 interior. This is expected behavior and is excluded from comparison.

## Key Findings for Compositor

1. **Zero poisoned tiles at R=850M.** The compositor can assume all TileOps are valid. No overflow fallback path is needed at this radius.

2. **Angle-invariant statistics.** All distributions — primes, groups, ports, payload budget — are statistically identical across 85°, 67.5°, and 45°. The compositor does not need angle-dependent parameters or band-specific handling.

3. **~10 groups, ~51 ports per tile.** Seam-merging between adjacent tiles will typically process ~13 port pairs per face (51 ports / 4 faces ≈ 12.8 per face). Max 22 groups is well under the 127 group-ID ceiling.

4. **Face symmetry.** All four faces carry ~12.8 ports on average. No face is structurally heavier than another. Symmetric compositor logic is correct.

5. **Budget headroom.** Mean slack is 48 bytes. Even at P99 (96 bytes used), there is 29 bytes of room. The 128-byte TileOp encoding is well-sized for R=850M and has margin for radius growth.

6. **Empty face edge case.** At 45°, Face I min = 0. The compositor must handle empty faces gracefully (zero-length port sections). This is not a correctness issue — the TileOp encodes empty faces correctly via off_I = off_L.

---

## Anomaly: Face I min = 0 at 45°

The 45° band contains at least one tile with zero inner-face ports. The inner face is the tile boundary closest to the origin.

At 45°, the arc curvature combined with the y=x diagonal slicing geometry can leave some tiles with no Gaussian primes on the inner boundary. The TileOp encodes this correctly: off_I = off_L (zero-length I section). This is a valid geometric edge case, not a pipeline bug.

The compositor must handle empty port sections on any face without special-casing the 45° band.
