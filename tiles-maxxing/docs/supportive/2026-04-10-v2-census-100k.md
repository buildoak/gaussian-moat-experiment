# TileOp V2 100K census at R=860M

## Scope

Command run:

```bash
./build/census_tiles 860000000 3125 --mt
```

This enumerates 3,125 towers x 32 tiles/tower = 100,000 tiles and measures the
post-prune TileOp V2 envelope for the current C++ pipeline.

Runtime on this machine: 99.7s wall clock, about 1003 tiles/s.

## Verdict

TileOp V2 with group-bit steal is sufficient at `R = 860,000,000` for the full
100K census that was requested here.

- Poisoned / overflow tiles: `0 / 100,000`
- Budget overflow tiles: `0`
- Group-cap overflow tiles (`group_count > 127`): `0`
- h1 encoding failure tiles (`h1 > 256`): `0`
- Max payload used: `114 / 125` bytes
- Min payload headroom: `11` bytes
- Max group count: `21 / 127`
- Group-cap headroom at the worst tile: `106`

At this operating point, the V2 envelope is not merely passing; it is passing
with substantial margin on both payload bytes and group count.

## Overflow / poison analysis

No poisoned tiles were observed in the full 100K run.

| Cause | Count |
|---|---:|
| Payload budget exceeded | 0 |

## Notes

- This requested census sweep includes tiles beginning at `a_lo = 0`, so it
  includes axis-touching towers. That matches the current tower generator used
  by `census_tiles.cpp`, but it is looser than the off-axis testing guidance in
  `AGENTS.md`. I did not alter the census domain because the task explicitly
  requested the 100K run at `R = 860M`.
- The updated census tool now records overflow causes, per-face counts,
  payload/headroom, `h1 = 256` usage, and multi-face-group diagnostics directly
  into the CSV, so future envelope checks at other radii can reuse the same
  instrumentation.
