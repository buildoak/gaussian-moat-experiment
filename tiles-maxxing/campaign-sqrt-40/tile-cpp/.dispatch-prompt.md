## Task: Compile and run tile-cpp at R~850M off-axis, sanity-check results

### Context
You are working with a C++ implementation of a Gaussian prime tile processor. The code lives at `tiles-maxxing/tile-cpp/`. It processes tiles (256x256 lattice windows) to find Gaussian primes, build connected components under step size k^2=40, extract boundary ports, and encode the result into a 128-byte TileOp.

The code has been tested only at small coordinates ((0,0), (100,100), (10000,10000)). We need to verify it works at the actual operating point: R~850 million, off-axis.

### What to do

1. **Create a CLI runner** -- Write a new `run_tile.cpp` in `tiles-maxxing/tile-cpp/tests/` that:
   - Takes tile coordinates (a_lo, b_lo) as command-line arguments (int64)
   - Calls `init_sieve_tables()`, `init_backward_offsets()`, `process_tile()`
   - Prints: prime_count, group_count, ports_before_pruning, ports_after_pruning
   - Prints wall-clock timing
   - Dumps the first 32 bytes of TileOp hex

2. **Update CMakeLists.txt** to build the new `run_tile` target

3. **Compile** in Release mode:
   ```bash
   cd tiles-maxxing/tile-cpp/build
   cmake .. -DCMAKE_BUILD_TYPE=Release
   make -j$(nproc)
   ```

4. **Run THREE off-axis tiles at R~850M:**
   - Tile A (45 degrees): a_lo=601040640, b_lo=601040640 (R ~ 850.0M)
   - Tile B (30 degrees): a_lo=736121088, b_lo=424999936 (R ~ 850.3M)
   - Tile C (15 degrees): a_lo=820888320, b_lo=220000000 (R ~ 849.9M)

   All coordinates are multiples of 256 (tile side).

5. **Sanity checks on each tile:**
   - prime_count > 0 (a 256x256 tile at 850M should have Gaussian primes)
   - group_count > 0 and <= prime_count
   - ports_before_pruning >= ports_after_pruning
   - No crashes, no buffer overflows, no assertion failures
   - Print TileOp hex prefix

6. **Cross-check with Python validator** -- In `tiles-maxxing/tile-validator/`:
   ```bash
   cd tiles-maxxing/tile-validator
   python3 -c "from tile import sieve_tile; primes = sieve_tile(601040640, 601040640); print('Python prime_count at (601040640,601040640):', len(primes))"
   ```
   Compare the Python prime_count with the C++ prime_count for Tile A. They MUST match exactly.

7. **Also run one SMALL tile** (e.g., (256, 256)) as baseline to confirm existing functionality and provide timing comparison.

### Key technical details
- TileCoord has fields: int64_t a_lo, int64_t b_lo -- bottom-left corner
- TILE_SIDE=256, COLLAR=7 (sieve window is 270x270)
- MAX_PRIMES=16384, MAX_PORTS=2048
- Norms at 850M are ~7.2e17 -- ALL norm arithmetic must be int64. Watch for int32 overflow.
- Sieve needs base primes up to sqrt(norm_max) ~ 850M -- memory-intensive, might need ~100MB
- If any tile takes >120s, flag it as a performance concern

Structured report:
