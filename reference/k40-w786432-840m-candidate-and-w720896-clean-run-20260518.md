## Claim Semantics

- Current detector verdicts are static-annulus `ANY-SPAN` results.
- `SPANNING` means some component connects `geo_I` to `geo_O` in the tested
  annulus.
- Timeout/non-finish rows are candidate pressure, not `MOAT`.
- A `MOAT` detector row requires full ingest and must later be hardened with
  audit telemetry plus sample postflight before acceptance.
- This clean run uses `telemetry=none`; it can localize pressure but cannot
  produce accepted/profile evidence by itself.

## Pre-Mortem

Failure modes and mitigations:

- Treating a long-running row as a moat claim.
  - This document labels long runtime as candidate pressure only.
- Running a BZ-invalid boundary.
  - Exact BZ check passed before launch and is stored in the run directory.
- Losing provenance.
  - The launch script writes `GENERATOR.txt`, `run-matrix.tsv`,
    `summary.tsv`, `driver.log`, BZ log, stdout/stderr, git commit/status, and
    GPU metadata under the run directory.
- Burning open-ended compute without supervision.
  - No internal timeout, but explicit human checkpoints at 2h, 6h, and 12h.
