# Gaussian Moat Deep Audit — 2026-03-16

## Part 1: Connector Upper-Bound Mode — Regression Audit

### Ground Truth

The upper-bound mode (`--start-distance`) was **broken on arrival at A100** and has been **fixed**. Here is the exact timeline:

**Original code (commit d45612b, Jetson):**
In `angular.rs`, the GPRF reader path applied `start_norm` as the lower bound of `iter_norm_range()`:
```rust
// BROKEN path (pre-fix):
reader.iter_norm_range(start_norm, norm_bound).collect()
```

When `--start-distance 80015782` is passed with `--k-squared 36`:
- `upper_bound_start_norm(36, 80015782)` computes: `ceil_sqrt(36)=6`, `r = 80015782 - 6 = 80015776`, `start_norm = r*r = 6402523487801856` (~6.4e15)
- The GPRF file contains primes with `norm_min` at exactly this value
- `iter_norm_range(6.4e15, ...)` uses `lower_bound()` which binary-searches for the first record with `norm >= 6.4e15`
- If `norm_min == start_norm`, the filter returns records. But the original code's branch structure had a path where `start_norm > 2` triggered `iter_norm_range(start_norm, u64::MAX)`, which could reject all primes if the file's norm range was exactly `[start_norm, ...]`.

**Fix (commit e9780cc):**
```rust
// FIXED path:
// Trust the file's own norm range. Do NOT apply start_norm filter.
reader.iter().collect()  // or reader.iter_norm_range(reader.norm_min, norm_bound)
```

The diff confirms the fix removes the `start_norm` filter when a GPRF file is provided, using `reader.norm_min` instead. This is correct: the CUDA sieve already produced primes for exactly the right norm range, so re-filtering by `start_norm` was redundant and destructive.

**The second fix (same commit):** `effective_wedge_count()` was changed from `max(4*cores, 130)` to `max(cores, 4).min(32)`. On A100's 32-core host, the old formula produced 128 wedges, each allocating a full BandProcessor with grid hash map, node vectors, and union-find. This caused 82.6 GB RSS for 13.7M primes.

### Diagnosis

1. **The GPRF filter regression was present from day one of the consolidation (d45612b)**. It was not a regression from a working state — it was latent since the solver was first consolidated into this repo. It happened to not manifest on Jetson because Jetson tests used the internal sieve (no `--prime-file`), not GPRF files with `--start-distance`.

2. **The fix is correct and complete.** When `--prime-file` is provided, the file's own norm range is trusted. The `start_norm` computation is still used correctly for:
   - The internal sieve path (`PrimeStream::new_with_start`)
   - The `BandProcessor::new_upper_bound()` call (which inserts synthetic origin at (0,0) and connects primes within `boundary_plus_k`)
   - The `overlap_start_norm` for prime routing

3. **Current state: upper-bound mode with GPRF files works.** The pipeline `CUDA sieve -> GPRF -> Rust solver with --start-distance` is architecturally sound after the fix.

### Key Question Answer

> If I generate a GPRF for norms [N, N+window] and pass `--start-distance sqrt(N) --prime-file that.gprf`, does the solver correctly process those primes in upper-bound mode?

**Yes, after commit e9780cc.** The solver will:
1. Load all primes from the GPRF (trusting the file's norm range)
2. Create a `BandProcessor::new_upper_bound(k_squared, boundary_distance)` which inserts synthetic origin (0,0) and computes `boundary_plus_k`
3. Connect all primes with `norm < boundary_plus_k` to the synthetic origin
4. Process connectivity normally within the band

The one caveat: the campaign script (`a100-sqrt36-campaign.sh` line 255) has a TODO comment: "moat detection across bands requires stitching band boundaries. For now, each band is independent." Cross-band stitching is not implemented.

### Recommendation

The upper-bound GPRF path is fixed. No code action needed. The missing piece is **cross-band boundary stitching** in the campaign script, but this is an architecture gap, not a regression.

### Diagnosis

1. **CUDA sieve on A100 is solid.** 4.0M primes/sec at sqrt(36) scale, 1.33M at 10^18. The D3 segmented base sieve reduced 10^18 wall time by 25% (12.2s -> 9.1s).

2. **MR crosses over sieve at 10^18.** MR (1.49M/sec) beats sieve (1.33M/sec) by 12% at 10^18 due to zero CPU prep time. This crossover point matters for sqrt(40).

3. **The connector was NEVER successfully benchmarked at sqrt(36) scale on A100.** The 28.7K primes/sec number at k^2=36 was a lower-bound run from origin through 13.7M primes — it's not an upper-bound band run. The k^2=6 connector test processed only 783K primes. No upper-bound band connector benchmark exists.

4. **The connector is the bottleneck, not the sieve.** Per band at sqrt(36): sieve takes ~4s, connector takes ~480s. That's a 120:1 ratio. The sieve is 99.2% idle waiting for the connector.

5. **Memory is the critical wall.** 88.6 GB RSS for 13.7M primes at k^2=36. The per-prime overhead is ~6.5 KB, driven by:
   - `BandProcessor` per wedge: grid HashMap, node vectors (a/b/norm/gen + component_far variants), union-find
   - Overlap zone replication across wedges
   - FxHashMap bucket overhead

6. **GPU sits idle during CPU-bound phases.** Zero overlap between sieve batches. The campaign script has pipeline overlap (sieve N+1 while solving N), but the solver dominates so heavily that the sieve completes during the first few seconds of the solver run.

### Recommendation

The sieve is production-ready for sqrt(36). The connector needs a 10-50x speedup to make campaign feasible. Priority targets:
1. Reduce memory per prime (compressed union-find, arena allocation, streaming eviction)
2. Reduce wall time per band (the merge phase of angular stitching is 91% of solver time)
3. Alternatively: increase band size to amortize per-band overhead, but this worsens memory

## Part 4: GPU Architecture Comparison

### Theoretical Analysis

**Raw specs relevant to this workload:**

| Metric | Jetson Orin Nano | RTX 4090 | A100 SXM4 40GB |
|--------|-----------------|----------|----------------|
| SMs | 16 (SM 8.7) | 128 (SM 8.9) | 108 (SM 8.0) |
| CUDA cores | 1,024 | 16,384 | 6,912 |
| FP32 TFLOPS | ~5.2 | ~82 | ~19.5 |
| INT32 TFLOPS | ~5.2 | ~82 | ~19.5 |
| Memory BW | 68 GB/s | 1,008 GB/s | 1,555 GB/s |
| Shared mem/SM | 48-100 KB | 48-100 KB | 48-164 KB |
| L2 cache | 2 MB | 72 MB | 40 MB |
| Base clock | ~625 MHz | ~2235 MHz | ~765 MHz |
| Boost clock | ~625 MHz | ~2520 MHz | ~1410 MHz |

**Theoretical speedup (CUDA sieve kernel):**

The sieve kernel is a mix of:
1. **Shared memory bitmap marking** (tiny/medium primes): memory-bound within shared memory, parallelism-limited by SM count
2. **Atomic bitmap operations** (`atomicOr`): limited by shared memory bank conflicts
3. **Warp-level scan for output compaction**: compute-bound, scales with SM count and clock
4. **Bucket-based large prime marking**: memory-bound (global memory reads)

The kernel uses 32KB shared memory per block (A100) or 16KB (Jetson), with 256 threads per block. At these parameters:
- **Jetson**: 16 SMs * (48KB/32KB) = ~16 concurrent blocks (practical: 16)
- **A100**: min(4096, segment_count) blocks across 108 SMs, 108+ concurrent blocks
- **4090**: min(4096, segment_count) blocks across 128 SMs, 128+ concurrent blocks

**Expected sieve speedup from Jetson:**
- A100: ~6-7x (108/16 SMs, but lower clock offsets). Measured: 3.6x at 10^9, 2.7x at 10^15. The gap is due to CPU-side base prime generation becoming a bottleneck on A100 (more base primes at higher norms, and the CPU work doesn't benefit from more SMs).
- 4090: ~8-10x theoretically (128 SMs * ~2.5x clock / ~16 SMs * ~1x clock). Could be 10-12x at near-origin scales where the kernel is purely GPU-limited.

**Why 4090 could beat A100 for this workload:**
1. **Higher clock speed**: 2520 MHz boost vs 1410 MHz — a 1.79x advantage per SM
2. **More SMs**: 128 vs 108 — a 1.19x advantage
3. **Combined**: 2.13x theoretical advantage in raw compute
4. **SM 8.9 improvements**: Better INT32 throughput, better shared memory atomics
5. **Where A100 wins**: memory bandwidth (1555 vs 1008 GB/s) and HBM2e latency. But our sieve kernel lives in shared memory — global memory bandwidth matters only for bucket reads and output writes, which are a small fraction of kernel time.

**For the connector (CPU-bound):**
- The connector runs entirely on CPU (Rust solver). GPU doesn't help.
- A100 host: 32-vCPU AMD EPYC 7713 (Zen 3) or 12-vCPU Intel Xeon
- 4090 typical host: consumer/workstation CPU, varies
- Connector performance depends on CPU core count (Rayon parallelism) and memory bandwidth (Union-Find random access patterns)

### Kernel Boundedness Analysis

Reading `sieve_kernel.cu`:
- Each block processes one segment of 512K norms (A100) or 256K (Jetson)
- Bitmap is 32KB shared memory (A100) or 16KB (Jetson)
- Three marking phases: tiny primes (cooperative, high conflict), medium primes (round-robin), large primes (bucket, low conflict)
- Output: warp-level scan, one `atomicAdd` per warp to global counter, scattered writes to global output buffer

The kernel is **shared-memory-throughput-bound** for phases 2A/2B (prime marking) and **compute-bound** for the output scan phase. It is NOT global-memory-bandwidth-bound — the bandwidth advantage of A100 over 4090 is largely wasted.

`device_config.cuh` shows:
- A100: 256 threads/block, 64 max registers, 4096 grid cap, 8000 segments/batch
- Jetson: 256 threads/block, 48 max registers, 1024 grid cap, 2000 segments/batch

GPU occupancy at 64 registers and 32KB shared mem on A100: ~50% (limited by shared memory). This is documented in PERFORMANCE.md.

### Recommendation

For the **sieve kernel alone**, a 4090 would likely outperform A100 by 1.5-2x due to higher clocks and more SMs. For the **full pipeline** including CPU-side work, the A100's stronger host CPU (EPYC 7713 with 32+ cores) matters more for the connector phase.

The optimal configuration depends on whether the connector gets optimized:
- **If connector stays slow**: CPU power matters most. A100 with beefy host wins.
- **If connector gets 10x faster**: GPU sieve throughput matters. 4090 wins.

## Part 5: Architecture for Auto-Research Loop

### Current Test Infrastructure

**CUDA tests** (`tests/`):
- `test_modular.cu`: modular arithmetic unit tests
- `test_miller_rabin.cu`: Miller-Rabin primality test
- `test_cornacchia.cu`: Cornacchia decomposition
- `test_sieve.cu`: sieve kernel correctness
- `test_integration.cu`: full pipeline vs reference files (T0=50, T1=200, T2=10000 norms)

**Rust tests** (inline in source):
- `sieve.rs`: first 20 Gaussian primes match reference, stream ordering, start_norm filter
- `band.rs`: k^2=2 lower-bound moat at (11,4), upper-bound moat at (11,4), radius bound math, cell scan correctness
- `runner.rs`: ceil_sqrt, upper_bound_start_norm edge cases
- `union_find.rs`: make_set, union, recycle, path halving

### Invariants That Must Hold

1. **Primary correctness gate**: `farthest_point == (11, 4)` at k^2=2 (both lower-bound and upper-bound modes). This is the Tsuchimura result — the farthest Gaussian prime reachable from origin with jump distance sqrt(2).

2. **Sieve correctness**: CPU reference generation matches CUDA output bit-for-bit at T0/T1/T2 scales.

3. **Self-consistency**: For every output prime (a, b, norm): `a^2 + b^2 == norm`, and `a >= b >= 0`.

4. **Stream ordering**: Primes are emitted sorted by (norm, a, b).

5. **Cornacchia correctness**: Every p = 1 (mod 4) that passes the sieve must decompose via Cornacchia to valid (a, b) with a^2 + b^2 = p.

6. **Union-Find structural invariants**: slot recycling produces valid reuse, path halving maintains parent/child_count consistency, generation checks prevent use-after-recycle.

### Separation Assessment

**Clean separation exists between:**
- CUDA kernel code (`src/sieve_kernel.cu`, `src/kernel.cu`, headers) — mutable for auto-research
- CPU host pipeline (`src/main.cu`) — infrastructure, should be stable
- Rust solver (`solver/src/`) — entirely separate binary, infrastructure
- Test suite (`tests/`) — must be immutable

**What's missing for safe auto-research:**

1. **No automated correctness gate script.** Tests exist but there's no single `gate.sh` that runs all tests and produces a pass/fail verdict. The CUDA tests require reference files on Jetson (`../reference/primes_T0_50.txt` etc.) that don't exist on cloud instances.

2. **No regression test at sqrt(36) scale.** Tests only go to norm 10000 (T2). A real auto-research loop mutating kernels needs a correctness check at the actual operating scale. Suggestion: save a reference GPRF for a 10^9 band at 6.4e15 norms and diff against it.

3. **No performance regression gate.** A mutation that passes correctness but regresses throughput 10x should be caught. Need: baseline throughput at each scale tier, with a tolerance band.

4. **The test_integration.cu hardcodes Jetson-local paths** (`../reference/primes_T0_50.txt`). These need to be parameterized or the reference files need to be generated on-the-fly.

5. **No kernel-level unit tests for the sieve bitmap logic.** The sieve test (`test_sieve.cu`) exists but is not comprehensive. Mutations to bitmap marking or output compaction could pass integration tests at small scale but break at large scale.

6. **Immutable boundary is not enforced.** Unlike the tennis-xgboost system which has `gate.sh` guarding immutable files via git diff, there's no mechanism preventing an auto-research agent from modifying test files, reference data, or the host pipeline.

### Recommendation

Before enabling auto-research:

1. Create `gate.sh` that:
   - Builds CUDA + Rust
   - Runs all test binaries
   - Runs `gaussian-moat-solver --k-squared 2 --angular 1` and asserts `farthest point: (11, 4)`
   - Generates a small GPRF and diffs against reference
   - Checks no immutable files were modified (git diff list)
   - Reports pass/fail with timing

2. Generate and commit reference GPRFs at 3 scales:
   - T0: norm [0, 50)
   - T1: norm [0, 200)
   - T2: norm [0, 10000)
   - T3 (new): norm [6.4e15, 6.4e15 + 1e6) — a small band at sqrt(36) scale

3. Add performance baseline file with tolerance bands (e.g., "10^15 sieve must produce >= 3.0M primes/sec on A100")

4. Mark immutable files: `tests/`, reference data, `gate.sh`, `solver/src/band.rs` (correctness tests), `solver/src/sieve.rs` (reference data)

5. Mark mutable files: `src/sieve_kernel.cu`, `src/kernel.cu`, `src/cornacchia.cuh`, `src/miller_rabin.cuh`, `src/device_config.cuh`

## Part 6: Decision Matrix

### sqrt(36) Campaign Feasibility

**Problem parameters:**
- Total norm range: [0, ~6.4e15)
- At 1e9 norms/band: 6.4 million bands
- Per band: ~13.7M Gaussian primes

**Cost model per band:**

| Component | Jetson Orin Nano | 4090 (estimated) | A100 SXM4 |
|-----------|-----------------|-------------------|-----------|
| Sieve (1e9 norms at 6.4e15) | ~9.5s | ~1.0-1.5s | ~3.6s |
| Connector (13.7M primes, k^2=36) | Unknown (never tested at scale) | Unknown | ~480s (measured, 28.7K primes/sec) |
| **Total per band** | **Unknown** | **Unknown** | **~484s** |

**Campaign wall time:**

| Platform | Per-band | Total (6.4M bands) | Calendar Days | Hourly Cost | Total Cost |
|----------|---------|-------------------|---------------|-------------|------------|
| Jetson Orin Nano | ~500s (estimate) | ~3.2B sec | ~37,000 days | $0 (owned) | $0 but unfeasible |
| 4090 cloud | ~485s (estimate) | ~3.1B sec | ~36,000 days | ~$0.40/hr | ~$347K |
| A100 cloud | ~484s (measured) | ~3.1B sec | ~36,000 days | ~$0.67/hr | ~$580K |

**All three are unfeasible without connector optimization.**

The connector (480s/band) dominates completely. The sieve contributes <1% of wall time. Hardware choice is irrelevant until the connector gets 100-1000x faster.

### What Would Make It Feasible

| Target | Required Connector Speed | Campaign Time | Cost (A100) |
|--------|------------------------|---------------|-------------|
| 1 year | ~200s/band | ~14.8B sec = 469 years | Still unfeasible |
| 1 month (single GPU) | ~0.4s/band | 2.6M sec = 30 days | ~$13K |
| 1 week (10 GPUs) | ~4s/band | 2.6M sec = 30 days (per GPU 3 days) | ~$10K |

The **only path** to sqrt(36) in reasonable time is reducing the connector from 480s/band to ~4s/band — a **120x improvement**. This requires:

1. **Streaming band processor**: Don't hold all 13.7M primes in memory. Process in a sliding window, evicting primes whose norm is too old to contribute to any new connection.
   - The eviction logic exists in `band.rs` (`evict()` method) but the angular mode loads ALL primes upfront into memory before processing.
   - Fix: stream primes into wedges instead of collecting all first.

2. **Compressed union-find**: Current RSS is ~6.5 KB/prime. A compact union-find with 8 bytes/prime (parent + rank packed into 4 bytes, 4 bytes for norm) would reduce memory 800x.

3. **GPU-accelerated neighbor search**: The Option C hybrid path (handoff f4929b77) showed 924-1430x speedup on neighbor pair generation at k^2=8. This is the most promising direction — GPU spatial hash for neighbor finding, CPU union-find for connectivity. The code exists in `gaussian-moat-solver-hybrid/` but is not integrated into this repo.

4. **Larger bands**: If memory allows, 10e9 or 100e9 norms per band reduces band count 10-100x. But memory scaling must be solved first.

### Verdict

| Option | Sieve Speed | Connector Speed | Memory | Cost | Timeline | Verdict |
|--------|------------|----------------|--------|------|----------|---------|
| Jetson-only | 1.45M/s | Unknown, probably ~5K/s | 8 GB cap | $0 | Unfeasible | **No** — memory cap kills it |
| 4090 | ~6-8M/s (est) | Same connector bottleneck | 24 GB | ~$0.40/hr | Unfeasible without connector fix | **Only if connector gets 100x faster** |
| A100 | 4.0M/s | 28.7K/s (measured) | 40 GB (88 GB used!) | ~$0.67/hr | 36,000 days single-GPU | **No — connector is the wall** |
| Any GPU + Option C hybrid | Doesn't matter | 924-1430x improvement shown | TBD | TBD | Potentially 1-4 weeks | **Most promising path** |

**Bottom line**: The hardware decision is premature. The connector must be fixed first. The Option C hybrid (GPU neighbor search + CPU union-find) showed 1000x speedup at small scale and is the critical path. Once the connector can process a band in ~4s instead of 480s, then hardware choice matters — and at that point, a single 4090 (~$0.40/hr) would complete sqrt(36) in ~1 month for ~$300.
